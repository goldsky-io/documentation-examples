Compose · Turbo · Mechanics Brief

Corporate-Actions Distributor: how it actually runs

What the components are, what runs when, and where the guarantees come from. Compose orchestrates Turbo as an ephemeral, on-demand subroutine. Live on Base mainnet.

The pieces

On-chain
DistributionCampaign.sol
Holds USDC escrow. declare() pulls funds and records the operator's chosen recordBlock; pay() releases per holder; paid[id][holder] map prevents double-pay.
On-chain
ShareToken (ERC-20)
The thing being distributed against. Emits standard Transfer events — that's all the indexer needs.
Compose · 1 app
corporate-actions
Two TypeScript tasks: declare_campaign (HTTP) and process_campaigns (cron · 1 min). Drives the lifecycle for every campaign.
Turbo · 1 per campaign
corp-actions-<id>
Job-mode pipeline. Spawned by compose at declare time; bounded by end_block: recordBlock; backfills, flushes the snapshot, exits. Compose deletes it on terminal.

What lives where

Key point: the Turbo pipeline only exists while a campaign is being snapshotted. Compose creates it, polls it, deletes it. No always-on indexing, no manual pipeline lifecycle to manage.

The flow, in order

1
Issuer
Deploys ShareToken. Constructor pre-mints to the 10 demo holders → 10 Transfer(0x0 → holder) events in the deploy block. (For a real share token, you skip this step.)
2
Operator
POST /tasks/declare_campaign with {campaignId, recordBlock, totalAmount}. The HTTP task validates recordBlock ≤ currentBlock, then: (a) USDC.approve(campaignContract, totalAmount), (b) DistributionCampaign.declare(...) which atomically pulls escrow, (c) POST /api/v1/pipelines to spawn a job-mode Turbo pipeline filtered to shareToken with end_block: recordBlock, (d) inserts a campaign row at status: "snapshotting".
3
Turbo
The pipeline backfills base.erc20_transfers, source-level filter on the share token (Fast Scan: the dataset returns only matching rows, not the whole chain). Decodes via the dataset schema, emits credit/debit deltas via SQL UNION ALL. The postgres_aggregate sink rolls deltas into share_balances_<id>. When it hits end_block, Turbo flushes the final checkpoint and transitions to completed.
4
Compose cron
Every minute. For each non-terminal campaign: polls GET /api/v1/pipelines/<name>/state every 5 seconds (up to 30 polls per tick). On completed → flips to status: "paying", drives the first payment tick immediately. On error → flips to failed, deletes the pipeline, drops the per-campaign tables.
5
Compose cron
Reads the snapshot from share_balances_<id>. Pro-rata: each holder gets floor(balance × totalAmount / totalSupply); the floor remainder is assigned to the last holder so the sum equals totalAmount exactly.
6
Compose cron
For each holder: readContract(isPaid(id, holder)). If true, skip. If false, writeContract(pay(id, holder, amount, sharesAtSnapshot)) via the Privy-managed operator wallet (sponsored gas). Bounded concurrency = 5. Promise.allSettled, so one revert doesn't block the others.
7
Compose cron
Re-reads on-chain isPaid for all holders. If all true → mark campaign complete, DELETE the pipeline, drop the per-campaign tables. Future ticks ignore it. Done.

Where the guarantees come from

GuaranteeEnforced by
No double-pay, everOn-chain require(!paid[id][holder]). The cron is just a driver — correctness lives in the contract.
Idempotent declareCompose collection unique key on userId. Duplicate POST returns the existing campaign without touching chain or spawning a second pipeline.
Snapshot at exact blockOperator passes recordBlock explicitly; the pipeline runs end_block: recordBlock; Turbo guarantees end_block data is durably flushed before the job exits. The contract also records the same block in declare() so the audit trail is self-consistent.
Crash recovery (snapshotting)The pipeline runs in Turbo's infrastructure, not the compose pod. A pod kill mid-snapshot is a no-op: the next cron tick polls /state cold and resumes.
Crash recovery (paying)Cron is stateless between ticks. Restart re-reads on-chain isPaid. Already-paid holders are skipped without an attempted tx.
Pro-rata sums to exact totalMath helper assigns floor remainder to the last holder (sorted lexicographically). No rounding drift.
No cross-campaign aggregate contaminationEach campaign's pipeline writes to its own share_balances_<id> table. Two concurrent campaigns can't sum into the same row.
No orphaned infrastructureCompose DELETEs the pipeline and drops the per-campaign tables on terminal. Belt-and-suspenders: even if Turbo's auto-cleanup runs first, the explicit DELETE is idempotent (a 404 is treated as success).
Operator never holds gasevm.wallet({sponsorGas: true}) → Privy-managed wallet with EIP-7702. Bundler pays gas, operator pays nothing.

Why this shape

One Turbo pipeline per campaign, ephemeral. Earlier iterations of this example ran an always-on streaming pipeline that maintained share_balances live. That works, but it forces operators to think about pipeline lifecycle (deploy, monitor, restart on schema change) for an indexer that's only needed during the brief window between declare and final payout. Job-mode + ephemeral inverts that: the operator declares an intent, compose handles every operational concern. Turbo becomes a callable subroutine, not a piece of infrastructure to babysit.

Operator-supplied recordBlock. Real corporate actions have a record date set in advance. The snapshot is by definition backwards-looking. Putting recordBlock in the request body matches that mental model directly — and it makes the snapshot deterministic: anyone can re-derive the holder list at the same block from public chain data.

Push-based vs pull-based. This pattern caps at ~100 holders before push throughput becomes the bottleneck. Production-scale distributions (Apple's millions of shareholders) use merkle-claim contracts where holders pull. The orchestration, idempotency, and audit-trail primitives shown here apply to either model — only the on-chain shape changes.
