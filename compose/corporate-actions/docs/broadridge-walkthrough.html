Compose · Turbo · Mechanics Brief

Corporate-Actions Distributor: how it actually runs

What the components are, what runs when, and where the guarantees come from. Compose orchestrates Turbo as an ephemeral, on-demand subroutine — one HTTP call drives the entire declare → snapshot → pay lifecycle. Live on Base mainnet.

The pieces

On-chain
DistributionCampaign.sol
Holds USDC escrow. declare() pulls funds and records the operator's chosen recordBlock; pay() releases per holder; paid[id][holder] map prevents double-pay; escrowRemaining goes to zero when every holder has been paid.
On-chain
ShareToken (ERC-20)
The thing being distributed against. Emits standard Transfer events — that's all the indexer needs.
Compose · 1 app
corporate-actions
Two TypeScript HTTP tasks: declare_campaign drives the full lifecycle in a single request; list_holders serves the dashboard's drill-down view.
Turbo · 1 per campaign
corp-actions-<id>
Job-mode pipeline. Spawned by compose at declare time; bounded by a filter-level block range from the share token's deploy block up to recordBlock; backfills, flushes the snapshot, exits. Compose deletes it on terminal.

What lives where

Key point: the Turbo pipeline only exists while a campaign is being snapshotted. declare_campaign creates it, polls it, drives the payouts, and deletes it — all inside a single HTTP request. No always-on indexing, no cron handoff, no manual pipeline lifecycle to manage.

The flow, in order

1
Issuer
Deploys ShareToken. Constructor pre-mints to the demo holder set (25 addresses, long-tailed amounts) → 25 Transfer(0x0 → holder) events in the deploy block. (For a real share token, you skip this step.)
2
Operator
POST /tasks/declare_campaign with {campaignId, recordBlock, totalAmount}. The task validates recordBlock ≤ currentBlock, then: (a) USDC.approve(campaignContract, totalAmount), (b) DistributionCampaign.declare(...) which atomically pulls escrow, (c) POST /api/v1/pipelines to spawn a job-mode Turbo pipeline filtered to shareToken over block_number BETWEEN deployBlock AND recordBlock, (d) inserts a campaign row at status: "snapshotting".
3
Turbo
The pipeline backfills base.erc20_transfers bounded by the filter's block range — the planner prunes everything before the deploy block and after the record block, so the scan only touches the window the share token has actually existed in. The postgres sink lands raw Transfer rows in share_balances_<id>. When the bounded scan finishes, Turbo flushes the final checkpoint and the deployment exits.
4
declare_campaign
Same request, still running. Polls GET /api/v1/pipelines/<name>/state every 2 seconds. On completed (or the auto-cleanup 404 that follows a successful job-mode run, paired with rows actually present in the agg table) → flips the campaign to status: "paying" and continues. On error → flips to failed, deletes the pipeline, drops the per-campaign tables.
5
declare_campaign
Reads the snapshot from share_balances_<id> via a SQL aggregate (credits to recipients, debits from senders, summed per account). Pro-rata: each holder gets floor(balance × totalAmount / totalSupply); the floor remainder is assigned to the last holder so the sum equals totalAmount exactly. Persists the full payouts list (holder, shares, amount) onto the campaign row so the dashboard's drill-down survives the per-campaign table cleanup.
6
declare_campaign
For each holder: readContract(isPaid(id, holder)). Already-paid holders are skipped. The remainder fire concurrently — up to 25 pay() calls in a single batch via the Privy-managed operator wallet (sponsored gas). Promise.allSettled, so one revert doesn't block the others. Each pay() tx hash is stitched back onto the persisted payout for basescan deep-linking.
7
declare_campaign
Reads campaigns(id).escrowRemaining on-chain. Zero → mark campaign complete, DELETE the pipeline, drop the per-campaign tables, return. Non-zero → leave the campaign in paying so a re-POST of the same campaignId resumes from where we left off.

Where the guarantees come from

GuaranteeEnforced by
No double-pay, everOn-chain require(!paid[id][holder]). The driver is just orchestration — correctness lives in the contract.
Idempotent declareCompose collection unique key on userId. A second POST with the same campaignId resumes the existing campaign without touching chain or spawning a second pipeline. Terminal states return immediately.
Snapshot at exact blockOperator passes recordBlock explicitly; the pipeline filter caps the scan at recordBlock; Turbo guarantees the data within the bounded range is durably flushed before the job exits. The contract also records the same block in declare() so the audit trail is self-consistent.
Crash recovery (snapshotting)The pipeline runs in Turbo's infrastructure, not the compose pod. A pod kill mid-snapshot is a no-op: the operator re-POSTs the same campaignId, the resume path polls /state cold against the still-running pipeline.
Crash recovery (paying)The driver is stateless across calls. Re-POST re-reads on-chain isPaid; already-paid holders are skipped without an attempted tx. The persisted payouts field on the campaign row reproduces the exact pro-rata that was originally computed, so resumed payouts match the original snapshot.
Pro-rata sums to exact totalMath helper assigns floor remainder to the last holder (sorted lexicographically). No rounding drift.
"Done" is unambiguousA single contract read of escrowRemaining decides terminality. The contract decrements it inside pay() with checks-effects-interactions, so reaching zero is a deterministic post-condition of the last successful pay() — not something the driver has to infer from N isPaid reads.
No cross-campaign aggregate contaminationEach campaign's pipeline writes to its own share_balances_<id> table. Two concurrent campaigns can't sum into the same row.
No orphaned infrastructureCompose DELETEs the pipeline and drops the per-campaign tables on terminal. Belt-and-suspenders: even if Turbo's auto-cleanup runs first, the explicit DELETE is idempotent (a 404 is treated as success).
Operator never holds gasevm.wallet({sponsorGas: true}) → Privy-managed wallet with EIP-7702. Bundler pays gas, operator pays nothing.
Drill-down survives cleanupPayouts (holder, shares, amount, pay tx hash) are persisted onto the campaign row in the compose collection — not just the per-campaign table. list_holders reads from there, so the dashboard's expanded view works long after the snapshot table has been dropped.

Why this shape

One Turbo pipeline per campaign, ephemeral. Earlier iterations of this example ran an always-on streaming pipeline that maintained share_balances live. That works, but it forces operators to think about pipeline lifecycle (deploy, monitor, restart on schema change) for an indexer that's only needed during the brief window between declare and final payout. Job-mode + ephemeral inverts that: the operator declares an intent, compose handles every operational concern. Turbo becomes a callable subroutine, not a piece of infrastructure to babysit.

Inline drive instead of cron. An earlier shape split lifecycle work across declare_campaign and a 1-minute process_campaigns cron. The cron handoff was correct but slow — the typical campaign sat idle waiting for the next minute boundary even though the snapshot was already done. Folding everything back into declare_campaign means the response to the operator's POST is the completion signal: when the request returns, the campaign is paid out. Resume-on-re-POST gives back the crash-recovery property that the cron used to provide, without the latency.

Operator-supplied recordBlock. Real corporate actions have a record date set in advance. The snapshot is by definition backwards-looking. Putting recordBlock in the request body matches that mental model directly — and it makes the snapshot deterministic: anyone can re-derive the holder list at the same block from public chain data.

Push-based vs pull-based. This pattern caps at ~100 holders before push throughput becomes the bottleneck. Production-scale distributions (Apple's millions of shareholders) use merkle-claim contracts where holders pull. The orchestration, idempotency, and audit-trail primitives shown here apply to either model — only the on-chain shape changes.
